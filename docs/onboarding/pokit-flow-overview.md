# POKit 흐름 한 장

POKit의 기본 흐름은 하나입니다.

```text
1. 요청
   "이런 걸 만들고 싶어"
        |
        v
2. 다듬기
   작업 후보 / 완료 기준 / 확인 질문
        |
        v
3. 이슈 선택
   지금 실행할 이슈 하나
        |
        v
4. 실행
   문서 작성, 코드 수정, 조사, 정리
        |
        v
5. 검증
   테스트, 자동 점검, 문서 확인, 필요한 증거
        |
        v
6. 게이트
   증거를 보고 완료 판정
        |
        v
7. 다음
   다음 이슈로 이동하거나 스프린트 종료
```

개념이 먼저 필요하다면 [pokit-concepts-for-users.md](./pokit-concepts-for-users.md)를 먼저 보세요.

## 설치와 시작

처음 시작할 때는 한 줄이면 됩니다.

```bash
npx pokit2 install
```

npm 계정이나 로그인은 필요 없습니다. 설치하면 프로젝트에 세 가지가 생깁니다.

```text
AGENTS.md           POKit 버전 정보(frontmatter 포함)
.claude/            얇은 스킬 모음
.ai-os/             이슈·상태 파일 (여기가 작업의 중심)
```

본체는 글로벌 한 벌로 관리됩니다. 프로젝트에는 상태와 규칙만 남습니다.

글로벌 설치를 선호한다면 `npm i -g pokit2` 후 `pokit install`로 동일하게 초기화할 수 있습니다.

## 주요 명령어

| 하고 싶은 것 | 명령 |
|---|---|
| 현재 상태 확인 | `pokit start` |
| 새 이슈 만들기 | `pokit issue:create --title "..."` |
| 이슈 목록 보기 | `pokit issues` |
| 프로젝트 점검 | `pokit doctor` |
| 본체 업데이트 | `pokit update` |

`pokit update`는 도구 소유 파일만 갱신합니다. 이슈와 상태(`.ai-os`)는 건드리지 않습니다.

## 검증과 게이트는 다릅니다

```text
검증 = 증거 모으기
게이트 = 그 증거로 통과 판정하기
```

예를 들어 테스트 통과와 자동 점검 통과는 검증입니다. 게이트는 그 검증 결과, 변경 내용, 완료 기준을 보고 "이 이슈는 닫아도 된다"고 판정하는 단계입니다.

## 프로젝트가 있을 때

프로젝트는 작업 공간입니다. 이슈 번호, 현재 상태, 다음 행동을 프로젝트별로 나눕니다.

```text
common / COM-001      기본 프로젝트
alpha  / ALP-001      별도 프로젝트
shop   / SHP-001      또 다른 프로젝트
```

프로젝트를 따로 만들지 않으면 기본은 `common`입니다. 여러 제품이나 고객 작업을 나눠야 할 때 새 프로젝트를 만들면 됩니다.

## 세션을 나눌 때

기본은 메인 세션 하나로 충분합니다. 동시에 다른 일을 하거나, 한 이슈를 별도 작업 공간에서 처리하고 싶을 때 작업 세션과 worktree를 씁니다.

```text
메인 / 통합 세션
  - 현재 상태와 게이트를 관리
  - 작업 세션 결과를 검토
  - 최종 반영과 완료 판정 담당

작업 세션
  - 별도 worktree에서 작업
  - 공유 상태를 직접 바꾸지 않음
  - 변경 제안으로 결과를 전달

overview
  - 여러 프로젝트 상태를 읽기만 함
  - 수정, push, 게이트 판정 없음
```

이 모델의 핵심은 간단합니다.

```text
나눠서 작업할 수는 있지만,
완료 판정은 증거를 보고 통합 지점에서 한다.
```

## 안내 카드: 각 단계의 길잡이

실행 흐름의 각 단계에서 러너는 안내 카드를 출력합니다.

```text
work 카드      지금 어떤 이슈를 작업 중인지, 다음 행동이 무엇인지
integration 카드   통합 검토가 필요한 상황과 판단 기준
status 카드    전반적인 현재 상태 요약
```

카드는 표시 전용입니다. 카드가 뜬다고 단계가 자동으로 승인되거나 게이트가 통과되지 않습니다.

## 안전 단계 자동화: 🟢 자동 / 🔴 사람 확인

모든 단계가 같은 방식으로 진행되지 않습니다.

```text
🟢 자동 진행 (되돌릴 수 있고 흔적이 남는 단계)
   코드 변경, 검증, 커밋

🔴 사람 확인 필요 (외부 영향이거나 완료 확정)
   push, 게이트 통과
```

push는 항상 PO 확인이 필요합니다. 🟢 단계라도 근거 없이 건너뛰지 않습니다.

## 이슈 없이 시작하면 막힙니다

활성 이슈가 없는 상태에서 파일을 바꾸려 하면 자동으로 차단됩니다.

```text
이슈 없이 파일 수정 시도
  -> 자동 차단
  -> 이슈 초안 자동 제안
  -> 이슈를 만들고 활성화한 뒤 다시 진행
```

이 안전 바닥은 이슈 ID 앞 글자(COM, POK 등)에 무관하게 동작합니다.

## 세션이 여러 개일 때

worktree나 세션을 여럿 열어도 작업 후보 중복 선택이 자동으로 방지됩니다.

```text
세션 A가 후보 X 작업 중
  -> 세션 B가 pokit issues 실행
  -> 세션 B는 후보 X를 받지 않음 (이미 사용 중)
  -> 세션 B는 다른 후보를 안내받음
```

세션은 시작 시 자동으로 등록됩니다. 로컬 훅은 advisory(--no-verify로 우회 가능)이며, 최종 강제 적용은 서버 측에서 이루어집니다.

## push는 자동으로 하지 않습니다

POKit은 작업을 합치거나 게이트를 통과했다고 해서 자동으로 외부에 push하지 않습니다. push는 별도의 정책이나 PO 확인이 필요합니다.

```text
작업 세션       push 안 함
통합/메인       정책 또는 PO 확인 후 push 가능
overview        읽기 전용
```

## 가장 작은 사용 흐름

```text
포킷 시작
  -> 현재 상태 확인

만들고 싶어 / 고치고 싶어
  -> 작업 후보와 완료 기준 정리

진행하자
  -> 이슈 실행

다음
  -> 게이트 통과 후 다음 이슈로 이동
```

처음에는 이 네 문장만 기억해도 됩니다.

## 파일로 남는 것

POKit은 중요한 상태를 파일로 남깁니다.

```text
.ai-os/current.md              지금 하는 일
.ai-os/status-board.md         짧은 현황
.ai-os/memory/session/handoff.md 다음 세션을 위한 인계
projects/<project>/issues/     이슈 카드
docs/                          문서 산출물
```

그래서 세션을 새로 열어도 "어디까지 했는지"를 다시 복구할 수 있습니다.

## Antigravity 런타임에서 쓸 때

POKit의 네 가지 기술(`/pokit.backlog`, `/pokit.clarify`, `/pokit.issue`, `/pokit.next`)은 Antigravity 런타임에서도 `define_subagent` 기반 에뮬레이션 계약으로 사용할 수 있습니다.

```text
진입점: ANTIGRAVITY.md
계약 방식: define_subagent 에뮬레이션
공식 지원 상태: deferred (스모크 통과, 전체 증거 미수집)
```

현재는 에뮬레이션 계약만 제공됩니다. Antigravity 런타임에서 네이티브 기술 자동 발견이 된다고 가정하지 마세요. 런타임별 증거가 생긴 뒤에 공식 지원으로 전환됩니다.
