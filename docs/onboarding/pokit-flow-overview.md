# POKit 흐름 한 장

POKit의 기본 흐름은 하나입니다.

```text
1. 요청
   "이런 걸 만들고 싶어"
        |
        v
2. 다듬기
   작업 후보 / 완료 기준 / 확인 질문
        |
        v
3. 이슈 선택
   지금 실행할 이슈 하나
        |
        v
4. 실행
   문서 작성, 코드 수정, 조사, 정리
        |
        v
5. 검증
   테스트, 자동 점검, 문서 확인, 필요한 증거
        |
        v
6. 게이트
   증거를 보고 완료 판정
        |
        v
7. 다음
   다음 이슈로 이동하거나 스프린트 종료
```

## 검증과 게이트는 다릅니다

```text
검증 = 증거 모으기
게이트 = 그 증거로 통과 판정하기
```

예를 들어 테스트 통과와 자동 점검 통과는 검증입니다. 게이트는 그 검증 결과, 변경 내용, 완료 기준을 보고 "이 이슈는 닫아도 된다"고 판정하는 단계입니다.

## 프로젝트가 있을 때

프로젝트는 작업 공간입니다. 이슈 번호, 현재 상태, 다음 행동을 프로젝트별로 나눕니다.

```text
common / COM-001      기본 프로젝트
alpha  / ALP-001      별도 프로젝트
shop   / SHP-001      또 다른 프로젝트
```

프로젝트를 따로 만들지 않으면 기본은 `common`입니다. 여러 제품이나 고객 작업을 나눠야 할 때 새 프로젝트를 만들면 됩니다.

## 세션을 나눌 때

기본은 메인 세션 하나로 충분합니다. 동시에 다른 일을 하거나, 한 이슈를 별도 작업 공간에서 처리하고 싶을 때 작업 세션과 worktree를 씁니다.

```text
메인 / 통합 세션
  - 현재 상태와 게이트를 관리
  - 작업 세션 결과를 검토
  - 최종 반영과 완료 판정 담당

작업 세션
  - 별도 worktree에서 작업
  - 공유 상태를 직접 바꾸지 않음
  - 변경 제안으로 결과를 전달

overview
  - 여러 프로젝트 상태를 읽기만 함
  - 수정, push, 게이트 판정 없음
```

이 모델의 핵심은 간단합니다.

```text
나눠서 작업할 수는 있지만,
완료 판정은 증거를 보고 통합 지점에서 한다.
```

## push는 자동으로 하지 않습니다

POKit은 작업을 합치거나 게이트를 통과했다고 해서 자동으로 외부에 push하지 않습니다. push는 별도의 정책이나 PO 확인이 필요합니다.

```text
작업 세션       push 안 함
통합/메인       정책 또는 PO 확인 후 push 가능
overview        읽기 전용
```

## 가장 작은 사용 흐름

```text
포킷 시작
  -> 현재 상태 확인

만들고 싶어 / 고치고 싶어
  -> 작업 후보와 완료 기준 정리

진행하자
  -> 이슈 실행

다음
  -> 게이트 통과 후 다음 이슈로 이동
```

처음에는 이 네 문장만 기억해도 됩니다.

## 파일로 남는 것

POKit은 중요한 상태를 파일로 남깁니다.

```text
.ai-os/current.md              지금 하는 일
.ai-os/status-board.md         짧은 현황
.ai-os/memory/session/handoff.md 다음 세션을 위한 인계
projects/<project>/issues/     이슈 카드
docs/                          문서 산출물
```

그래서 세션을 새로 열어도 "어디까지 했는지"를 다시 복구할 수 있습니다.
